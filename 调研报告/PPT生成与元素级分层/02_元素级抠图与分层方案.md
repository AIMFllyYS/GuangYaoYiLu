# 元素级抠图与分层方案调研

## 1. “元素”的严格定义

在 PowerPoint 语境里，元素不是视觉区域，而是 slide shape tree 里的对象。python-pptx 文档说明，几乎所有出现在 slide 上的东西都是 shape；每张 slide 有一个 shape tree。常见对象类型包括：

- 文本框和普通几何形状：`<p:sp>`
- 图片：`<p:pic>`
- 分组：`<p:grpSp>`
- 连接线：`<p:cxnSp>`
- 表格、图表等 graphic frame：`<p:graphicFrame>`
- content part：`<p:contentPart>`

所以“按元素抠图”的正确目标是：按这些对象或对象组提取，而不是按画面上的固定横纵区域切块。

## 2. 当前 v3/v4 分层不是元素级

当前 `交付物/答辩PPT/答辩PPT_v4/layer_manifest.json` 的统计结果：

- slides：22
- total layers：180
- min/max layers per slide：7/11
- role 分布：
  - `ai_tile`：132
  - `official_logo`：22
  - `real_photo`：26
- 高频层名：
  - `header`：22
  - `left_field`：22
  - `center_field`：22
  - `right_field`：22
  - `footer_left`：22
  - `footer_right`：22
  - `hust_logo`：22

这说明当前分层主要是“整页 AI 图按固定版面区域切片”，只有 logo 和真实照片是相对接近语义元素的对象。`header`、`left_field`、`center_field`、`right_field` 不是 PowerPoint 元素，它们只是视觉区域。

## 3. 为什么从整页 PNG 无法可靠恢复元素

如果输入只有一张完整 slide PNG：

- 文本框的边界已经被 rasterize。
- 图片和背景可能融合在同一像素层。
- 形状、阴影、渐变、装饰纹理没有对象 ID。
- z-order 丢失。
- 字体、字号、段落、换行规则丢失。
- 分组关系丢失。
- 表格和图表的结构数据丢失。

视觉模型可以猜测“这里像标题”“这里像照片”“这里像装饰”，但这只是视觉分割，不是原生元素恢复。尤其遇到国风纹理、水墨背景、装饰线条、半透明遮罩时，模型很容易切出无意义区域。

## 4. 真正可行的三条路线

### 路线 A：有源 PPTX，解析 shape tree

如果已有 PPTX 源文件，应直接读取每页 slide XML：

- 遍历 `ppt/slides/slideN.xml` 的 `<p:spTree>`。
- 按 XML 元素类型识别 `sp`、`pic`、`graphicFrame`、`grpSp`、`cxnSp`。
- 提取每个对象的 bbox：`a:xfrm/a:off` 和 `a:xfrm/a:ext`。
- 提取 z-order：shape tree 中的顺序。
- 提取 name/id：`p:cNvPr id/name`。
- 图片通过 slide rels 找 `r:embed` 对应 media。
- 文本从 `p:txBody` 提取。
- 表格从 `a:tbl` 提取。
- 图表从 chart relationship 提取。

这条路线最适合“按元素抠图”和“元素级动画”。

### 路线 B：从生成源头保留元素

如果 PPT 是 Codex 生成的，最优做法不是先生成整页图，而是在生成时就保留元素：

- 标题是 native text。
- 正文是 native text。
- 照片是 native picture。
- logo 是 native picture。
- 装饰线条/色块是 native shape。
- 表格是 native table 或可编辑 grid/text/shape 组合。
- 图表是 native chart。
- 每个关键对象有稳定 name，例如 `slide-title`、`subtitle`、`photo-01`、`hust-logo`、`chart-main`。

Codex Presentations skill 的 Granola JSX 正是这个方向：用 `text`、`image`、`shape`、`chart`、`table` 等 primitive 生成 PPTX-safe native content，并要求 key nodes 命名以便 layout inspection。

### 路线 C：只有图片时，只能做近似视觉切分

如果只有整页 PNG，可以做的是：

- OCR 找文本块。
- 图像显著性/边缘/连通域找图片区。
- 颜色和纹理找背景层。
- 人工规则保护 logo、照片、页码。
- 输出为“视觉候选层”，明确标注不等于 PowerPoint 原生元素。

这条路线只能作为补救，不适合承诺“按元素抠图”。要降低随机分组，必须让模型按 manifest schema 输出候选层，并人工/程序校验 bbox、重叠、z-order、命名。

## 5. 推荐的元素 manifest schema

后续如果重建，应输出类似结构：

```json
{
  "slide": 1,
  "elements": [
    {
      "id": "slide-title",
      "type": "text",
      "source": "native",
      "bbox": { "x": 120, "y": 96, "w": 920, "h": 96 },
      "z": 20,
      "text": "光药医路",
      "editable": true,
      "animation_group": "title"
    },
    {
      "id": "hust-logo",
      "type": "image",
      "source": "official_asset",
      "bbox": { "x": 42, "y": 22, "w": 260, "h": 61 },
      "z": 30,
      "editable": false,
      "animation_group": "brand"
    }
  ]
}
```

关键点：

- `type` 必须是语义对象，不是 `left_field` 这种区域名。
- `source` 区分 native、official_asset、real_photo、generated_background、visual_guess。
- `editable` 明确是否可编辑。
- `bbox` 和 `z` 是后续动画、导出、重建的基础。
- 如果是视觉猜测，必须显式标注 `source: visual_guess`。

## 6. 后续实践原则

- 能用原生元素就不用整页 PNG。
- 能解析源 PPTX 就不做截图视觉猜测。
- 背景可以是整页图，但标题、正文、logo、照片、表格、图表要独立。
- 动画 target 必须指向稳定对象 id/name。
- 分层报告必须区分“PowerPoint 元素”和“视觉切片”。
