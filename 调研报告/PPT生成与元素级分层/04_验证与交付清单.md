# PPTX 验证与交付清单

## 1. 分层验证模型

后续任何 PPTX 交付都至少分四层验证：

1. 文件层：ZIP、CRC、文件大小、图片可读。
2. Package 层：content types、relationships、part 存在性。
3. OpenXML schema 层：OpenXmlValidator 或等价工具。
4. 应用层：PowerPoint 或等价 headless import/render 打开 saved PPTX。

当前项目只稳定覆盖了第 1 层和第 2 层的一部分，因此不能继续把“validate 通过”写成最终安全结论。

## 2. 当前 validator 保留项

继续保留并扩展 `工具/validate_pptx_package.py`：

- ZIP CRC 检查。
- 必要 part 存在检查。
- relationship target 检查。
- slide 数量检查。
- docProps Slides 数量检查。
- 媒体图片 verify。

建议新增：

- 检查 `ppt/slideMasters/` 至少一个。
- 检查 `ppt/slideLayouts/` 至少一个。
- 检查 `ppt/theme/theme1.xml`。
- 检查 `ppt/presProps.xml`。
- 检查每个 slide 是否引用 slide layout，或明确记录“无 layout 的非标准简化包”。
- 检查 slide XML 是否 XML parse 成功。
- 检查 `p:cNvPr id` 是否唯一。
- 检查动画 target id 是否存在。
- 检查是否有外部 relationship。

## 3. OpenXmlValidator 阶段

需要 .NET 环境。当前机器 `dotnet` 不存在，所以这是未满足项。

建议后续增加一个小型验证项目或脚本：

```powershell
dotnet run --project tools/OpenXmlValidate -- "答辩PPT_vNext/xxx.pptx"
```

输出要求：

- 错误数量。
- 每条错误所在 part。
- XPath/path。
- 错误描述。
- 是否阻塞交付。

OpenXmlValidator 通过仍不等于 PowerPoint 完全不修复，但它比当前 package smoke test 强很多。

## 4. saved-PPTX parity 阶段

至少选择一种：

- PowerPoint 桌面人工打开，确认无修复弹窗。
- PowerPoint 打开后另存为，比较修复前后 package 差异。
- LibreOffice headless 转 PDF，确认命令成功并人工检查 PDF。
- Codex Presentations runtime 的 `Presentation.load/inspect/export` 或等价 saved-PPTX 重新导入。

当前机器没有 `soffice` / `libreoffice`，所以无法直接做 LibreOffice headless 检查。

## 5. 视觉与文字验证

如果继续做中文 PPT，必须保持：

- UTF-8 写入。
- 不用默认系统编码写中文。
- 对 `.md`、`.json`、`.xml` 做 mojibake scan。
- 检查 replacement character、Latin-1 mojibake 前缀、Cyrillic mojibake 前缀、中文常见 mojibake 词等标记。
- 检查文字是否被 rasterize；如果要求可编辑，文字必须是 native text。
- 检查标题、正文、页码、logo、图片不越界。

## 6. 元素级交付清单

如果目标是“按元素抠图/分层”，每页必须交付：

- 原生 PPTX。
- 元素 manifest。
- 每个元素的 type、bbox、z-order、source、editable、animation_group。
- 原图 preview。
- 元素 overlay preview。
- 与 PPTX shape tree 对齐的检查报告。

禁止把以下内容称为元素级：

- `header`
- `left_field`
- `center_field`
- `right_field`
- `footer_left`
- `footer_right`
- 任何纯粹按网格或区域切出来的 tile

它们只能叫视觉切片或区域层。
