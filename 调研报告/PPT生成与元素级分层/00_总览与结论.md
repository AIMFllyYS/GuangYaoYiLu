# PPT 生成损坏与元素级分层调研总览

日期：2026-05-09  
范围：仅调研与报告整理，不继续完善 PPT 视觉内容。

## 结论摘要

当前项目里多版 Codex 生成的 PPTX 存在一个共同特征：文件是用 Python 手写 ZIP 包和 PresentationML XML 拼出来的，而不是由 PowerPoint、Open XML SDK、python-pptx、PptxGenJS 或 Codex Presentations runtime 这类成熟生成器输出。它们能通过当前 `validate_pptx_package.py` 的浅层检查，但这个检查只覆盖 ZIP、关系目标、媒体图片和页数，不等于 PowerPoint 兼容。

打开后提示“PPT 损坏/需要修复”的最可能根因不是中文编码，也不是图片本身，而是 OOXML 包结构和 PowerPoint 语义兼容性不足。尤其是当前 PPTX 缺少常见且规范要求中重要的 master、layout、theme、presentation properties 等部分，且 v2 以后手写了动画 timing XML，这些都比普通图片和文本更容易触发 PowerPoint 修复逻辑。

“按元素抠图”的正确做法也不是把页面切成 header、left、center、right 等区域。PowerPoint 的元素是 slide shape tree 里的对象，例如文本框、图片、形状、表格、图表和分组。只有从 PPTX 原生结构或生成源头保留这些对象，才能做到真正按元素处理；如果只剩整页 PNG 或 AI 生成图，元素语义已经丢失，只能做不可靠的视觉分割。

## 最重要的执行建议

1. 停止把“手写完整 PPTX 包”作为主路径。后续要么使用干净 PowerPoint 模板作为母体，要么使用成熟 PPTX 库或 Codex Presentations runtime。
2. 先保证静态 PPTX 完整可打开，再谈动画。动画 timing XML 不应继续手写推进。
3. 元素级处理必须从原生元素开始：文本框、图片、形状、表格、图表分别建模并命名。
4. 当前 `validate_pptx_package.py` 只能作为第一关，不能作为“PowerPoint 不会报错”的证明。
5. 关键交付必须包含 PowerPoint 或等价 headless 渲染的 saved-PPTX parity 检查；当前机器缺 `dotnet` 和 LibreOffice，这一点是明确验证缺口。
